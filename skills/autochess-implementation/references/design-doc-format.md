# レシピ: 設計書 DESIGN_<task-id>.md を書く

設計担当（Claude）が実装担当（Codex / Claude Code）に渡す設計書の形式。
`docs/COLLAB_PROTOCOL.md §2.2` の正式テンプレ。実装が迷わないように、影響範囲・受け入れ基準・
壊してはいけない既存 API を必ず明記する。

## 置き場所・命名
- `docs/DESIGN_<task-id>.md`（例: `DESIGN_R1-meta.md`, `DESIGN_skill_overhaul.md`）。
- task-id は `docs/RELEASE_PLAN.md` のタスク表と対応させる。

## テンプレート

```markdown
# <task-id>: <タイトル>

> 設計: Claude / 実装: Codex・Claude Code / <日付>
> 依存: <先に必要なタスク> / 関連: <関連docへのリンク>

## ゴール
（1〜2文。何が達成されたら完了か）

## 影響範囲
- 新規ファイル: ...
- 変更ファイル: ...（行数感）
- 既存依存API（壊さないこと）: ...   ← 重要。既存の公開メソッド名を列挙

## 現状の把握（実装済みの土台）
（既存コードのどこを拡張するか。関数名・フィールド名を具体的に）

## データ構造
（必要なら C# クラス/構造体の skeleton を疑似コードで）

## 振る舞いの仕様
（具体的なフローとエッジケース。対象選択・タイミング・JA/EN）

## 受け入れ基準
- [ ] テストケース1（観測可能な形で）
- [ ] ...
- [ ] Compilation completed (Errors: False)

## 実装ヒント
- 既存の <Helper名> を雛形に再利用できる
- <ハマりどころ> に注意

## 未決事項（Codex への質問）
- なし／（あれば列挙。曖昧点は QUESTIONS.md にも）
```

## レビュー時の追記
Codex のコミットを diff で確認したら、同じ DESIGN ファイルの末尾に節を足す:

```markdown
## Review (YYYY-MM-DD, Claude) — ✅承認 / 要修正
- 良い点 / 設計どおりか / デグレ有無
- 受け入れ基準の充足状況
- balance や将来課題があれば RELEASE_PLAN / ROADMAP へ送る
```

承認したら `ROADMAP.md` の進行メモにも1行残す（真実のソースの同期）。

## ポイント
- 「既存依存API（壊さないこと）」を必ず書く。これが共有 switch / ヘルパのデグレを防ぐ。
- 受け入れ基準は**観測可能**に（「正しく動く」ではなく「再起動後に章ベストが残る」など）。
- 数値は仮で良いが、その旨を書き、最終調整は R3-balance に委ねると明記する。
